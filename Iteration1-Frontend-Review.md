# Iteration 1 Frontend Review

## 1. 文档目的

本文档用于总结 ATM 系统第一次迭代中前端模块的实现过程、调试结论、运行方式与后续建议，便于组内同步、阶段汇报和后续迭代衔接。

---

## 2. 迭代目标回顾

根据任务文档，第一次迭代中前端模块的目标为：

- 搭建 Vue 3 前端项目基础结构
- 配置路由、状态管理和接口请求封装
- 设计 ATM 欢迎页、登录页、主菜单页
- 实现登录页面与查询余额页面原型
- 为取款、存款、转账、修改密码、凭条等后续功能预留入口

本次迭代重点不是完成全部交易流程，而是建立一个可运行、可扩展、可联调的前端基础版本。

---

## 3. 本次完成内容

### 3.1 工程搭建

已在仓库下新增 `frontend/` 目录，并完成前端基础工程初始化。

技术栈如下：

- Vue 3
- Vite

### 3.2 页面与交互

本轮已完成以下页面：

- 欢迎页
- 登录页
- 主菜单页
- 查询余额页
- 功能占位页

其中：

- 欢迎页用于进入 ATM 自助终端流程

### 3.3 状态与接口设计

本轮已完成以下前端基础能力：

- 通过 Pinia 管理 ATM 会话信息
- 支持会话写入本地存储
- 路由守卫控制未登录访问
- 基于 OpenAPI 文档设计登录、退出、查询余额、查询账户信息请求
- 支持 `mock` 模式，便于后端未完成时先进行前端演示和流程验证

## 4. 调试与验证过程

### 4.1 初始验证

在完成文件搭建后，先对所有 `.js` 文件执行了语法检查：

```bash
node --check
```

这一步用于排除基础语法错误，保证前端结构和模块引用没有明显问题。

### 4.2 登录页校验问题修复

调试过程中发现登录表单校验返回值处理不严谨，成功校验可能被误判为失败，因此对登录页的表单验证逻辑进行了修正，确保在校验通过后能够正常发起登录请求。

### 4.3 余额页展示方式调整

最初余额页使用了 `Element Plus` 的统计组件方案。考虑到不同版本组件行为和样式插槽兼容性，为减少不必要的不确定性，后续改为更直接的卡片式展示方式，使余额页更稳定、更容易联调。

### 4.4 依赖安装与构建验证

前端依赖安装过程并非一次完成，中间出现了安装进程残留与半安装状态，直接表现为执行构建时提示：

```bash
vite: not found
```

后续重新确认依赖目录和锁文件状态后，补全安装并再次执行构建，最终构建通过。

### 4.5 开发服务启动问题

在当前执行环境中，首次启动开发服务时出现端口监听权限问题：

```bash
listen EPERM: operation not permitted 0.0.0.0:5173
```

在放开本地端口监听后，开发服务启动成功，可正常通过浏览器访问。

### 4.6 最终验证结果

本轮最终完成了以下验证：

- 前端依赖安装完成
- `npm run build` 构建通过
- `npm run dev` 开发服务成功启动
- 本地访问地址可用：`http://localhost:5173/`

## 5. 当前运行步骤

在 `frontend/` 目录下执行：

```bash
npm install
npm run build
npm run dev
```

浏览器访问：

```text
http://localhost:5173/
```

如果后端接口尚未就绪，可通过环境变量启用 mock 模式：

```bash
VITE_USE_MOCK=true
```

演示账号如下：

- 卡号：`6222020000000001`
- 密码：`123456`

如果后端接口已经可用，则可配置：

```bash
VITE_API_BASE_URL=http://localhost:8080
```

## 6. 当前代码评审结论

### 6.1 正面结论

本次前端第一次迭代已经满足任务文档对阶段交付物的要求，具备以下特点：

- 工程结构清晰，便于多人协作和后续扩展
- 页面流程完整，已经能演示 ATM 的核心入口路径
- 登录与余额查询具备联调基础
- 对未完成模块采用占位设计，避免主流程断裂
- 支持 mock 模式，降低对后端进度的依赖

整体判断：本轮前端结果适合作为第一次迭代交付版本。

### 6.2 风险与不足

当前仍有几点需要注意：

- 本地存储读取缺少异常兜底，如果存储内容损坏，理论上可能影响启动
- 当前主包体积偏大，构建时已出现 chunk size warning
- 登录输入校验仍可继续收紧，例如限制密码必须为 6 位数字
- 尚未完成真实后端接口的完整联调和异常分支验证

这些问题不影响第一次迭代交付，但建议在第二次迭代开始前优先处理。

---

## 7. 后续建议

针对下一次迭代，建议优先推进以下事项：

- 补充本地存储读取失败时的容错处理
- 优化 `Element Plus` 引入方式，减少首屏打包体积
- 与后端完成登录、余额查询接口的真实联调
- 开始实现取款、存款、转账、修改密码页面
- 为交易结果、错误提示、异常状态增加更清晰的用户反馈

如果从课程作业展示角度考虑，也建议将前端复盘、后端复盘和 UML 文档说明分开管理，文件名显式带上模块前缀，这样在答辩或汇报时会更清楚。
